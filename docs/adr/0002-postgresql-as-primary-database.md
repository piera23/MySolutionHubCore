# ADR-0002: PostgreSQL as the Primary Database Engine

**Date:** 2026-03-09
**Status:** Accepted

---

## Context

The initial prototype used SQLite (easy local setup) and Microsoft SQL Server was also considered. We needed to select a production database that supports the platform's requirements: full ACID transactions, robust JSON support, UUID generation, fine-grained index control, and an active open-source ecosystem.

---

## Decision

We adopt **PostgreSQL 16+** as the sole database engine for all environments (development, staging, production).

EF Core uses the **Npgsql.EntityFrameworkCore.PostgreSQL** provider. Hangfire uses **Hangfire.PostgreSql** with Npgsql connections. Docker development uses `postgres:16-alpine`.

All SQLite and SQL Server packages have been removed from the solution.

---

## Consequences

**Positive:**
- **Feature parity across environments.** Development, CI, and production all run against PostgreSQL, eliminating "works on my machine" database bugs.
- **Advanced PostgreSQL features available.** `gen_random_uuid()`, partial indexes (`WHERE column IS NOT NULL`), `GENERATED BY DEFAULT AS IDENTITY`, `pg_isready` for health checks, and `ON DELETE SET NULL` / `CASCADE` with full FK enforcement.
- **Single vendor.** One connection pooling configuration (Npgsql), one migration provider, one Hangfire storage adapter.
- **Open source and self-hostable.** No SQL Server licensing cost. Compatible with managed services (AWS RDS, Azure Database for PostgreSQL, Supabase).

**Negative:**
- **Local setup more involved than SQLite.** Resolved by providing a Docker Compose file that starts PostgreSQL with a single command.
- **EF Core InMemory provider used in unit tests.** The InMemory provider does not support `ExecuteUpdateAsync` (bulk updates). Tests that exercise those code paths must run against a real PostgreSQL instance (integration tests). This is an accepted trade-off documented in the test project.
- **Npgsql identity annotation in migrations.** EF Core migration files must use `NpgsqlValueGenerationStrategy.IdentityByDefaultColumn` instead of `Sqlite:Autoincrement`. A migration audit was performed and all seven migration files were updated.

---

## Alternatives Rejected

**SQLite** was rejected for production because:
- No concurrent write support (`SQLITE_BUSY` errors under load).
- Limited support for `ALTER TABLE` operations needed for EF migrations.
- No network access — incompatible with containerised multi-instance deployments.

**SQL Server** was rejected because:
- Licensing cost (SQL Server Standard/Enterprise).
- Windows-centric tooling (although Linux containers are supported, the ecosystem is heavier).
- Hangfire.PostgreSql is maintained closer to the Hangfire core than Hangfire.SqlServer.
